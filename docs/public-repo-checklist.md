# 공개 레포 전환 체크리스트

이 문서는 현재 `Mublo Framework` 저장소를 공개용 레포로 전환할 때 필요한 작업을 정리한 실행 체크리스트다.

## 목표

- 민감 정보가 없는 상태로 공개한다.
- 공개 대상과 비공개 대상을 저장소 구조상 명확히 분리한다.
- 외부 사용자가 설치, 테스트, 기여를 바로 시작할 수 있게 한다.
- 공개 후 이슈 폭증을 막기 위해 문서와 정책을 최소 기준 이상으로 맞춘다.

## 1. 즉시 차단 항목

### 1-1. 비밀값 제거

- [ ] 저장소 전체에서 API 키, 토큰, 패스워드, 개인 URL, 내부 계정 정보를 전수 스캔한다.
- [ ] 추적 중인 민감 파일을 제거하거나 샘플 파일로 치환한다.
- [ ] 이미 노출된 키는 서비스 측에서 폐기하고 재발급한다.
- [ ] Git 히스토리에도 민감 정보가 남아 있으면 rewrite 도구로 정리한다.

예시 점검 항목:

- 샘플 설정 파일에 실제 서비스 키가 들어 있지 않은지 확인한다.

완료 기준:

- 작업 트리와 Git 히스토리 기준으로 민감 문자열이 더 이상 검색되지 않는다.
- 운영 중인 외부 키는 모두 재발급 완료 상태다.

### 1-2. 공개 범위와 실제 추적 파일 정렬

- [ ] 공개 대상 모듈 목록을 확정한다.
- [ ] 비공개 또는 내부용 모듈은 별도 private repo, archive 브랜치, 사내 패키지 중 한 곳으로 이동한다.
- [ ] `.gitignore` 에만 의존하지 말고, 이미 추적 중인 파일은 실제로 index 에서 제거한다.

공개 범위 예시:

- Package: `Board`, `Shop`
- Plugin: `Banner`, `Faq`, `MemberPoint`, `Popup`, `SnsLogin`, `Survey`, `VisitorStats`, `Widget`

비공개 또는 별도 분리 대상 예시:

- 내부 운영 모듈
- 사내 전용 플러그인
- 실험용 패키지
- 리팩토링 전 백업성 디렉토리

완료 기준:

- 공개 대상 외 디렉토리가 기본 브랜치에 남아 있지 않다.
- README, 문서, 실제 디렉토리 구조가 같은 범위를 설명한다.

## 2. 라이선스와 공개 정책

### 2-1. 라이선스 단일화

- [ ] 루트 `LICENSE`, `README`, `composer.json`, 패키지별 `README`, 패키지별 `composer.json` 의 라이선스 표현을 일치시킨다.
- [ ] 서드파티 번들 파일의 라이선스 고지 방식도 검토한다.

점검 예시:

- 루트 `LICENSE`
- 루트 `README`
- 루트 `composer.json`
- 패키지별 `composer.json`

완료 기준:

- 모든 진입 문서에서 같은 라이선스를 안내한다.

### 2-2. 지원 범위 명시

- [ ] 무엇을 공식 지원하는지 적는다.
- [ ] 무엇이 예시 코드, 선택 모듈, 후속 공개 후보인지 적는다.
- [ ] 보안 이슈 접수 채널과 공개 이슈 구분 기준을 적는다.

권장 문서:

- `SECURITY.md`
- `SUPPORT.md` 또는 README 내 지원 범위 섹션

## 3. 설치 가능 상태 만들기

### 3-1. 설치 문서 검증

- [ ] 빈 환경에서 문서만 보고 설치가 가능한지 새 머신 기준으로 검증한다.
- [ ] PHP 버전, DB 버전, 필수 확장, 웹서버 예시를 한 곳에 모은다.
- [ ] `.env.example` 와 설치 가이드 예시가 실제 코드와 맞는지 확인한다.

예시 점검:

- `README` 와 `CONTRIBUTING` 의 PHP 버전이 일치하는지 확인한다.

완료 기준:

- 요구사항 문서가 하나의 기준으로 통일된다.
- 신규 사용자가 로컬 설치 후 `/install` 과 `/admin` 진입까지 재현 가능하다.

### 3-2. 트러블슈팅 최소 채우기

- [ ] 설치 실패
- [ ] DB 연결 실패
- [ ] `storage/` 권한 문제
- [ ] 관리자 로그인 실패
- [ ] 메일 발송 실패
- [ ] Redis 설정 오류

완료 기준:

- 가장 흔한 5~6개 장애에 대해 원인과 조치가 적혀 있다.

## 4. 품질 기준 정리

### 4-1. 테스트와 CI

- [ ] 공개 범위 기준 핵심 테스트 세트를 정의한다.
- [ ] `composer test` 와 CI가 같은 기준으로 돌도록 맞춘다.
- [ ] 선택 모듈 때문에 CI가 흔들리지 않게 테스트 범위를 분리한다.

완료 기준:

- 공개 브랜치에서 기본 테스트가 안정적으로 통과한다.

### 4-2. 코드베이스 정리

- [ ] 백업 파일, 임시 파일, 개인 작업 산출물, 발표 자료를 제거한다.
- [ ] 공개 브랜치에 불필요한 워크트리 산출물이 없는지 확인한다.
- [ ] 문서 오탈자와 내부 용어를 정리한다.

점검 대상 예시:

- `.clone/`
- `views/Admin/basic_backup/`
- `public/assets/lib/editor/mublo-editor/*.bak*`
- 루트의 개인 문서 파일 (`.pdf`, `.docx`, `.pptx`)

완료 기준:

- 저장소 루트와 공개 모듈 안에 개인/백업/임시 산출물이 없다.

## 5. 공개 커뮤니티 준비

### 5-1. 기여 흐름 보강

- [ ] 브랜치 전략
- [ ] 커밋 메시지 규칙
- [ ] PR 템플릿
- [ ] 이슈 템플릿
- [ ] 코드 리뷰 기준

추가 권장:

- `.github/ISSUE_TEMPLATE/`
- `.github/pull_request_template.md`

### 5-2. 첫인상 개선

- [ ] README 첫 화면에 설치, 문서 시작점, 공개 범위, 라이선스를 명확히 둔다.
- [ ] 스크린샷 또는 예시 사이트 설명을 최소 1개 넣는다.
- [ ] "무엇을 할 수 있는지"와 "무엇은 아직 안 되는지"를 분리해서 적는다.

## 6. 권장 작업 순서

1. 민감 정보 제거와 키 폐기
2. 공개 대상/비공개 대상 확정
3. 비공개 모듈 실제 분리
4. 라이선스와 정책 문서 정리
5. 설치 문서와 트러블슈팅 보강
6. 테스트/CI 기준 고정
7. README 최종 정리 후 공개

## 7. 공개 직전 최종 점검

- [ ] `git grep` 기준 민감 정보가 없다.
- [ ] 공개 대상 외 디렉토리가 기본 브랜치에 없다.
- [ ] 라이선스 표기가 전 문서에서 일치한다.
- [ ] 설치부터 로그인까지 문서 기반 재현이 된다.
- [ ] 핵심 테스트가 통과한다.
- [ ] README 첫 화면만 읽어도 사용자가 시작할 수 있다.
